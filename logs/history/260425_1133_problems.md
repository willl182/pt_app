# Problem Record: Participant `u_i` Must Not Fall Back to `sd_value`

**Severity**: High

## The Problem

The participant uncertainty flow previously allowed missing `u_i` to fall back to `sd_value`. That is methodologically wrong for `zeta` and `En`: the participant supplies its own uncertainty budget, and the app cannot infer it from repeated-result dispersion.

## User-Requested Corrections

- If the participant does not supply `u_i`, do not calculate `zeta` or `En`.
- Do not put `u_i` into `summary_n13.csv`; it remains in a separate participant PT data file.
- Rename the participant uncertainty file/source from `uncertainty` to `pt_data`.
- In reports, show participant `u_i` as the participant uncertainty and `u_xpt` as the assigned-value uncertainty.
- Treat `uncertainty_std` as an internal alias only; the conceptual value is `u_i`.

## The Solution

- Renamed the source file to `data/pt_data_n13.csv`.
- Replaced `uncertainty_df()` with `pt_data_df()` in `app.R`.
- Removed all fallback behavior that used `sd_value` as `uncertainty_std`.
- Set `uncertainty_std = u_i` when present and `NA_real_` when absent.
- Updated validation scripts to use `DATA_PT_DATA` and leave `zeta`/`En` non-calculable without `u_i`.
- Updated report table to use `p_data$u_i` for participant uncertainty and retain `p_data$u_xpt` for assigned-value uncertainty.
- Updated the plan file to clarify that `summary_n13.csv` does not contain `u_i`.

## Learnings to Preserve

- `u_i` is participant-reported and mandatory for `zeta` and `En`.
- `sd_value` must not be used as a substitute for participant uncertainty.
- `sd_value / sqrt(3)` is only a consistency check, never a replacement for `u_i`.
- The app may calculate `z` and `z_prime` without `u_i`, but not `zeta` or `En`.
- If the internal name `uncertainty_std` remains, document it as an alias for `u_i`; consider renaming it in a separate scoped refactor.
