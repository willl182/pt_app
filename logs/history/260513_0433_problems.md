# Problem Record (Troubleshooting)

**Severity**: Medium

## The Problem

The validation pipeline could not be re-run end-to-end with the requested
inputs out of the box.

Observed failures:

- R wrapper aborted in stage 1 while building the comparison table because a
  scalar `if (...)` received `NA`.
- Python wrapper reached stage 5, then failed on comparison-row key handling
  because the expected `section` field was missing in one path.
- The original pipeline was also coupled to `participants_data4.csv`, which
  does not match the requested `summary_n13.csv` run.

## User-Requested Corrections

- Re-run validation using:
  - `data/homogeneity - homogeneity.csv`
  - `data/stability - stability.csv`
  - `data/summary_n13.csv`
- Provide semiautomatic R and Python execution paths.
- Run each case separately with a subagent.

## The Solution / Workaround

- Added `scripts/run_validation_semiauto.R`.
- Added `scripts/run_validation_semiauto.py`.
- The wrappers back up the fixed validation inputs, copy the requested CSVs
  into place, run the validation stages, and restore the originals.
- Updated stage 5 to use `data/pt_data_n13.csv` for the participant budget
  source.
- Patched R stage 1 comparison logic to reduce failure risk on missing values.

## Learnings to Preserve

- `validation_1` is not fully data-parameterized; wrappers are safer than
  changing scientific logic for one-off replays.
- Python stage 5 and R stage 1 both have comparison-layer assumptions that
  need explicit guardrails for `NA` / missing fields.
- For `summary_n13.csv`, the participant-budget companion file must be the
  `n13` variant, not the `n4` one.
